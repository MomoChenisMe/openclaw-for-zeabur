# Project AI Memory

## 核心原則（最高優先級）

### 禁止過度設計

- 只做被要求的事，不「順便」改進周圍文件
- Bug 修復不需要重構周圍文件
- 一次性操作不需要 helper/utility/abstraction
- 不為假設性的未來需求設計

### 禁止過度防禦

- 不為不可能發生的情境添加錯誤處理
- 信任內部文件和框架保證
- 只在系統邊界驗證（用戶輸入、外部 API）

### 禁止未驗證的文件

- 不假設 API 簽名或可用方法
- 修改前必須先讀取檔案
- 用戶提及特定檔案/路徑時，必須先開啟檢查

### 禁止過早宣稱成功

- ❌ 「完美！問題已解決！」
- ✅ 「這個修改可能解決問題，讓我們測試看看...」

## 溝通原則

語言：臺灣繁體中文
態度：平等對話，避免過度正式
風格：直接、清晰、有邏輯
立場：有信心時保持技術立場

用語建議：

- ✅ 「這可能是問題所在...」
- ✅ 「我們可以試試...」
- ✅ 「這可能可以解決...」
- ❌ 「完美！」（驗證前）
- ❌ 「問題已解決！」（驗證前）

## 任務前檢查清單

執行任何任務前確認：

1. [ ] 這會過度設計嗎？（保持簡單）
2. [ ] 需求清楚嗎？（不清楚就問用戶）
3. [ ] 使用正確的工具嗎？（優先內建工具）
4. [ ] Commit message 符合標準？
5. [ ] 使用繁體中文溝通？
6. [ ] 語氣適當？（避免過早宣稱成功）

## 常見錯誤與修正

問題：創建不必要的抽象，添加未被要求的功能
修正：保持簡單，只實作必要需求

問題：git commit -m "update code"
修正：git commit -m "refactor(api): simplify customer query logic"

問題：「完美！問題已解決！」
修正：「這個修改可能解決問題，讓我們測試看看...」

問題：在需求模糊的情況下寫文件
修正：使用 AskUserQuestion 釐清需求
